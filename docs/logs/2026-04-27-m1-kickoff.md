# 2026-04-27 M1 Kickoff

## Purpose

`pdk-cartographer` begins as a Sky130-oriented PDK exploration atlas, but M1 deliberately uses synthetic fixtures. The purpose is to build durable foundations for learning Liberty timing structure, standard-cell metadata, and PDK anatomy without overstating what has been ingested or validated.

## Why This Follows CurveCraft

CurveCraft explored compact-model fitting and ngspice validation from the analog/device-model side. This project follows that work by moving into digital PDK artifacts: cell libraries, timing metadata, and the file anatomy behind ASIC implementation flows. The projects are separate, and CurveCraft remains untouched.

## M1 Decisions

- Fixture-first: start with tiny synthetic Liberty files before real Sky130 ingestion.
- Python 3.11+: keep the runtime modern but accessible.
- `src` layout: use `src/pdk_cartographer` for a clean import boundary.
- Quality gates: require pytest, ruff, and mypy as early as possible.
- No CLI in M1: keep the first milestone focused on package structure, fixtures, parser behavior, and docs.
- No Sky130 download in M1: avoid licensing confusion, repository bloat, and premature claims.

## Risks

- The parser could grow beyond its fixture purpose if scope is not enforced.
- Documentation could accidentally sound like real Sky130 analysis before ingestion exists.
- Future real-PDK ingestion will need careful read-only handling and citation of source data.

## Next Steps

- Finish M1 audit with validation commands and internal consistency checks.
- In M2, improve parser clarity and error messages while staying within a documented Liberty subset.
- Defer real Sky130 Liberty ingestion until the parser and reports have stable boundaries.
